# Architecture & Technical Design Document
## Hybrid Green Container Framework (HGCF)

### 1. System Architecture Overview

The Hybrid Green Container Framework (HGCF) follows a strictly decentralized, decoupled, and modular architecture. It is deployed as a standalone, privileged Docker container that observes, predicts, and shapes other containers running on the same host daemon.

The system interacts directly with the Linux kernel (via `/proc` and `cgroups`) and the Docker Daemon API (via `docker.sock`) to continuously evaluate container performance and adjust constraints.

### 3. Layered Design Breakdown

#### Layer 1: Environment Profiler (`framework/profiler.py`)
- Responsible for parsing system hardware context directly from the Linux `/proc` filesystem.
- Determines baseline limits for the shaping modules.
- Enforces an initial *Fallback Tier* policy for newly initialized nodes missing historical data.

#### Layer 2: Monitoring Engine (`framework/monitor.py`)
- Connects to the Docker daemon using the Python Docker SDK.
- Translates nested JSON `stats()` responses into flat, usable metrics (CPU %, Mem %, Mem Usage).
- **Self-Regulating Heuristics:** Alters polling velocity based on system pressure (10 seconds polling when active, 30 seconds when resting).

#### Layer 3: Hybrid Control Engine
This layer acts as the centralized brain for heuristic decisions.
- **Guardrail (`framework/guardrail.py`):** Holds a tiny volatile queue of the last 5 intervals. Instantly triggers aggressive throttling if bounds are broken.
- **Tier Detector (`framework/tier_detector.py`):** Manages a 120-tick sliding window leveraging `numpy.percentile()`. Dynamically labels workloads as Aggressive, Balanced, or Soft.
- **Predictor (`framework/predictor.py`):** Implements Adaptive Filter Moving Average (AFMV). Uses standard deviation logic to increase alpha variables during spiky intervals to heavily weight recent behavior.

#### Layer 4: Adaptive Resource Shaping (`framework/shaper.py`)
- Implements output mechanisms by generating Docker API update sequences.
- Calculates microsecond allocations (`cpu_quota`) relative to standard `cpu_period` logic defined within Linux cgroups.

### 4. Supplementary Services

#### Energy & Carbon Evaluator (`framework/energy.py`)
- Acts asynchronously during the logging pipeline.
- Translates utilization percentages into Joule/Watt projections based on known physical constants of the CPU model (Intel i3-4130 baseline).
- Converts energy output into granular carbon equivalents.

#### Metrics Visualization (`dashboard.py`)
- An independent lightweight Flask server exposed on port 8092.
- Ingests `metrics.csv` iteratively generated by the HGCF core loop.
- Visually plots live aggregations, averages, and container utilization without retaining massive databases.

### 5. Technical Stack
- **Core Language:** Python 3.10+
- **Math/Stats:** Numpy
- **API Interfacing:** Docker SDK for Python (`docker`)
- **Reporting UI:** Flask, HTML5, Chart.js, Bootstrap 5
- **Load Testing (Internal CI/Evaluation):** Locust
- **Deployment:** Docker, Docker Compose

### 6. Infrastructure & Deployment Model
HGCF is orchestrated strictly via `docker-compose.yml`.
- **Privilege Requirements:** `privileged: true` is set, and `pid: host` is supplied.
- **Volume Mounts:**
  - `/var/run/docker.sock` (API Access)
  - `/sys/fs/cgroup` (Read-Write for CPU Quota shaping)
  - `/proc/cpuinfo` & `/proc/meminfo` (Read-only for Profiling)
- **Container Networking:** Connects internally to the same network (e.g., `portfolio-network`) as the load generators and targets for complete observability.
